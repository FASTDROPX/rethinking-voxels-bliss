WHAT THIS BUILD ADDS
--------------------
A new cloud mode "Bliss" (CLOUD_STYLE 2), selectable alongside RV's
existing Reimagined (1) / Unbound (3) / Vanilla (50) / OFF (0). It renders
a 3-layer volumetric cloudscape (two cumulus decks + one high altostratus
sheet) inside RV's pipeline, lit by RV's own cloud light/ambient colours
and exposed through RV's native GetVolumetricClouds() interface so it
appears in the main sky view AND in water/ice/metal reflections with no
reflection-specific wiring.

=======================================================================
 HOW IT PLUGS INTO RV  (interface, unchanged since the first build)
=======================================================================
  shaders/lib/atmospherics/clouds/bliss_clouds.glsl
      - defines GetVolumetricClouds(...) with the EXACT signature RV's
        mainClouds.glsl already calls (so deferred1 + reflections.glsl +
        gbuffers_water + dh_water all drive it).
  shaders/lib/atmospherics/clouds/bliss_clouds_compat.glsl
      - supplies the Bliss #defines / atmosphere constants / light vectors
        that RV does not expose (WsunVec, sunElevation, ray-march counts,
        SKY_GROUND, CloudLayer toggles, etc.).
  shaders/lib/atmospherics/clouds/mainClouds.glsl
      - #ifdef CLOUDS_BLISS -> #include bliss_clouds.glsl
  shaders/lib/common.glsl, shaders.properties, lang/en_US.lang
      - CLOUD_STYLE value 2 = Bliss + the in-game Bliss cloud options
        (per-layer coverage/density/height, Cloud_Speed, Rain_coverage,
         BLISS_CLOUD_BRIGHTNESS).

Light/sky colour is bridged from RV (cloudLightColor / cloudAmbientColor *
BLISS_CLOUD_BRIGHTNESS). Bliss' own physical-sky LUT is NOT ported.

=======================================================================
 NOISE BRIDGE  (foundation; stable since Iteration 2)
=======================================================================
Bliss authored its cloud field against its own 512x512 RGBA noises.png
(3D-from-2D channel trick, hard-coded "/512" texel math). RV ships a
different noisetex (128x128 RGB). Sampling RV's texture with Bliss' texel
math is what caused the original blocky / tiling / moire regressions, so
the noise is reconstructed PROCEDURALLY and self-contained inside
bliss_clouds.glsl (Dave-Hoskins hashes -> quintic value noise -> fbm).
This never tiles and does NOT touch RV's noisetex, so RV's water and other
noise users are unaffected. These helpers are the foundation the density
field below is built on.

=======================================================================
 DEBUG ITERATION 4 -- DENSITY FIELD REWRITTEN FROM SCRATCH
=======================================================================
WHY THE PREVIOUS NOISE/COVERAGE FAILED TO STRETCH HORIZONTALLY
-------------------------------------------------------------
Bliss' original coverage used a billow transform on a texture read,
    coverage = abs(CloudLarge*2 - 1.2)*0.5 - (1 - CloudSmall)
    layer    = coverage + LAYER_COVERAGE
fed by 2D coords like worldXZ/20000 and worldXZ/2000. Two problems made it
impossible to get sweeping pancakes out of it with procedural noise:

  1. NO LARGE-SCALE TERM. With procedural noise sitting at its 0.5 mean,
     the billow evaluates to a near-constant ~0.3 everywhere -> a uniform
     full-sky sheet. There was nothing controlling WHERE clouds are vs
     open sky, so nothing could "stretch": every direction looked the same
     and at the horizon all of it integrated into a solid line.

  2. EROSION FREQUENCY TOO HIGH. The erosion (densityAtPos) effectively
     ran at ~1.5-block horizontal features. A near-vertical view ray
     crossing a thin layer in ~10 steps cannot sample 1.5-block detail, so
     it aliased into the radial "spikes / thorns" seen overhead. Adding a
     mask (iter 3) did nothing about this.

WHAT WAS CHANGED (exact math)
-----------------------------
The functions bliss_densityAtPos, bliss_cloudCov, bliss_cloudVol and the
old bliss_GetAltostratusDensity were DELETED. The cumulus and altostratus
density are now computed directly:

  * QUINTIC SHAPING CURVE (new bliss_quintic):
        q(x) = x*x*x*(x*(x*6 - 15) + 10)        // 6x^5 - 15x^4 + 10x^3
    Used for EVERY soft ramp / contrast step below. There is no step() and
    no smoothstep() anywhere in the density path -- only this explicit
    quintic. (The value-noise interpolation in bliss_vnoise2/3 also uses
    this exact polynomial on each axis.)

  * HORIZONTAL COVERAGE -- MASSIVE STRETCH (new bliss_coverageField):
        p   = worldXZ * BLISS_COVER_SCALE;   // 0.00045  -> 1 feature ~= 2200 blocks
        p.x *= BLISS_ANISO;                  // 0.55     -> ~4000-block bands along X
        cov = bliss_quintic( bliss_fbm2(p) );// 3 octaves, contrast-shaped
    The key change is BLISS_COVER_SCALE: multiplying world XZ by a tiny
    number stretches one noise feature across thousands of blocks, so the
    masses are huge and sweeping; BLISS_ANISO stretches them further along
    X into elongated pancake bands. The quintic makes them distinct from
    the gaps without a hard threshold.

  * SOFT COVERAGE ONSET (in bliss_GetCumulusDensity), menu-driven:
        layerCov = (layer==1) ? LAYER1_COVERAGE : LAYER0_COVERAGE; // in-game slider
        covAmt   = clamp(layerCov * BLISS_COVER_GAIN, 0, 1);       // 0.7*0.72 ~= 0.5
        d        = (cov - (1 - covAmt)) / BLISS_EDGE;
        cloud    = bliss_quintic( clamp(d, 0, 1) );                // NO hard step
    Below covAmt the field is open sky; above it the quintic ramps cloud in
    smoothly. Lower covAmt (or the in-game coverage slider) = more open sky.

  * VERTICAL VOLUME -- the "don't flatten to a 2D plane" fix:
        h        = (pos.y - minHeight) / (maxHeight - minHeight); // 0..1 in layer
        baseRamp = bliss_quintic( h / BLISS_BASE_FRAC );          // soft flat base
        topTaper = bliss_quintic( (1 - h) / BLISS_TOP_FRAC );     // soft rounded top
        cloud   *= baseRamp * topTaper;
    Density is shaped through the FULL thickness of the layer with smooth
    quintic attenuation strictly from the layer height limits, giving real
    3D volume instead of an infinitely thin sheet on the horizon. The three
    layers stay at their separate heights (250 / 500 / 2000, each 100 thick).

  * EROSION -- LOW FREQUENCY + MULTIPLICATIVE (kills the spikes):
        ep    = pos * BLISS_ERODE_SCALE;  // 0.006 -> ~165-block detail (was ~1.5)
        ep.xz *= 0.5;                      // stretched horizontally too
        er    = bliss_quintic( bliss_fbm3(ep) );
        cloud *= 1 - er * BLISS_ERODE_AMOUNT * (1 - cloud);
    Erosion features are now ~100x larger, so the view ray never
    undersamples them -> no aliased spikes. It is MULTIPLICATIVE and weighted
    by (1-cloud), so it only nibbles cloud EDGES (cores stay solid) and can
    never push density negative (no max()/clamp step needed for it).

  * ALTOSTRATUS (bliss_GetAltostratusDensity) rewritten the same way: one
    very large sweeping bliss_fbm2 field (BLISS_ALTO_SCALE 0.00018) with a
    quintic onset driven by the in-game CloudLayer2 coverage slider.

The result is driven entirely by clear, documented controls instead of
Bliss' texture-tuned constants.

TUNABLES (top of bliss_clouds.glsl)
   BLISS_COVER_SCALE   0.00045  smaller = bigger / more sweeping masses
   BLISS_ANISO         0.55     smaller = longer horizontal pancake bands
   BLISS_COVER_GAIN    0.72     scales the in-game coverage slider (open sky)
   BLISS_EDGE          0.30     soft-onset width (bigger = softer edges)
   BLISS_FBM_OCTAVES   3        coverage detail octaves
   BLISS_ERODE_SCALE   0.006    erosion detail size (do NOT raise -> spikes)
   BLISS_ERODE_AMOUNT  0.55     how deeply erosion bites the edges
   BLISS_BASE_FRAC     0.22     soft base ramp (fraction of layer height)
   BLISS_TOP_FRAC      0.45     soft top taper (fraction of layer height)
   BLISS_ALTO_SCALE    0.00018  altostratus mass size

=======================================================================
 DEBUG ITERATION 5 -- depth occlusion, fluff, world-anchor verification
=======================================================================
The Iteration 4 rewrite removed the spikes but exposed/!left three issues.
Each is addressed below; all changes are inside bliss_clouds.glsl.

(1) "CLOUDS FOLLOW THE CAMERA" -- diagnosed, root cause was NOT view-space
-------------------------------------------------------------------------
Checked the coordinate chain end to end:
    bliss_layerStartingPosition():
        position = dV_view*dither + cameraPos + (dV_view/abs(dV_view.y))*flip
    renderLayer(): rayProgress starts there and does rayProgress += dV_view
    GetCumulusDensity()/coverageField() sample pos.xz == rayProgress.xz
cameraPos IS cameraPosition (absolute world), so the sampled position is
absolute WORLD space already -- the noise is world-anchored, not camera
space. Injecting cameraPosition again would DOUBLE it and actually break
anchoring, so that was not done.

The real reason it LOOKED locked: BLISS_COVER_SCALE was 0.00045, i.e. cloud
features ~2200 blocks wide. Moving/looking shifts the sampled noise by a
tiny fraction of a feature, so the near-uniform field appears static. Fix:
   - BLISS_COVER_SCALE 0.00045 -> 0.0010  (~1000-block masses: real shape,
     visible motion parallax as the player moves)
   - BLISS_ANISO       0.55    -> 0.70    (rounder masses, less "tube")
   - plus the fluff pass in (3), whose fine detail makes the world-anchored
     parallax obvious.

(2) CLOUDS DRAWING THROUGH TERRAIN / MOUNTAINS -- real bug, now fixed
--------------------------------------------------------------------
Bliss' terrain-intersection test was compiled out (CLOUDS_INTERSECT_TERRAIN
undefined) AND it derived its cutoff from length(FragPosition), which in
this port is a normalized direction * 1024 (a constant ~1024) -- useless.
So nothing stopped the raymarch at solid geometry and clouds painted over
distant mountains.

RV already provides the right value: lViewPosM = world distance to the
solid fragment on this pixel (it is a parameter of GetVolumetricClouds and
is how RV's own Reimagined/Unbound clouds occlude -- see reimaginedClouds
lines "if (lTracePos > lViewPosM)"). Wiring:
   - new runtime global  float bliss_terrainDist;
   - adapter sets it:  bliss_terrainDist = (skyFade >= 0.7) ? 1e9 : lViewPosM;
       * skyFade >= 0.7 is RV's own "this pixel is sky" threshold -> 1e9 so
         distant horizon clouds are NOT culled.
       * otherwise it is the real terrain distance.
   - renderLayer (both the cumulus loop and the altostratus branch) now does
         IntersecTerrain = length(rayProgress - cameraPosition) > bliss_terrainDist;
     and breaks/returns when true. The old #ifdef CLOUDS_INTERSECT_TERRAIN
     blocks and the bogus length(FragPosition) cutoff were deleted.
Result: on terrain pixels the ray stops at the terrain depth, so clouds no
longer render over or through blocks/mountains; on sky pixels they render
to the horizon as before.

(3) "SMOOTH PLASTIC" -- added a high-frequency fluff pass (anti-aliased)
-----------------------------------------------------------------------
The 165-block erosion alone left smooth, synthetic surfaces. Added a second
detail pass in bliss_GetCumulusDensity (LoD>=1 only):
    fp     = pos * BLISS_FLUFF_SCALE; fp.xz *= 0.6;          // ~22-76 block detail
    fluff  = bliss_vnoise3(fp)*0.65 + bliss_vnoise3(fp*2 + 7.3)*0.35; // 2 octaves
    cloud *= 1 - bliss_quintic(fluff) * BLISS_FLUFF_AMOUNT * fluffFade * (1 - cloud);
Two things keep this from bringing the spikes back:
   - it is MULTIPLICATIVE and weighted by (1-cloud), so it only nibbles
     EDGES (cores stay solid) and can never go negative; and
   - fluffFade = quintic(clamp((bliss_rayVerticality - 0.30)/0.40, 0, 1)).
     bliss_rayVerticality = abs(view ray .y) is set in renderClouds. The
     fluff is therefore full strength only when looking up (small, densely
     sampled ray steps) and fades to ZERO toward the horizon (huge steps),
     which is precisely where fine detail used to alias into spikes. Fine
     detail is added exactly where the ray can resolve it.

NEW / CHANGED TUNABLES

   BLISS_ANISO        0.55    -> 0.70      rounder, less tube-like
   BLISS_FLUFF_SCALE  (new) 0.022          micro-detail size
   BLISS_FLUFF_AMOUNT (new) 0.55           fluff bite depth
   runtime globals: bliss_terrainDist, bliss_rayVerticality

=======================================================================
 DEBUG ITERATION 6 -- BILLOW CONTRAST MATH RESTORED (cells, not sheets)
=======================================================================
WHAT WAS WRONG
--------------
Iteration 4/5 replaced Bliss' coverage with a smooth, CONTINUOUS quintic
onset of a low-frequency fbm. Continuous noise + a soft onset can only ever
produce smooth, sinuous SHEETS -- there is no mechanism to fold the field
into discrete cells. On top of that the feature scale was guessed far too
large (~1000-block masses), so within any view the field barely varied =
"plastic". Both are fixed here.

CHANGE 1 -- RESTORED THE BLISS BILLOW COVERAGE (the abs() contrast math)
-----------------------------------------------------------------------
The smooth onset / vertical-profile / coverageField functions were DELETED
and Bliss' ORIGINAL density field was put back VERBATIM:
    bliss_densityAtPos, bliss_cloudCov, bliss_cloudVol, bliss_GetCumulusDensity
    bliss_GetAltostratusDensity
Coverage is once again the BILLOW form
    layer0:  abs(CloudLarge*2.0 - 1.2)*0.5 - (1.0 - CloudSmall) + LAYER0_COVERAGE
    layer1:  abs(CloudLarge - 0.8)         -        CloudSmall  + LAYER1_COVERAGE - 0.5
plus Bliss' Topshape/Baseshape vertical erosion and the Perlin-worley
cloudVol erosion. Taking the ABSOLUTE value of the (inverted) noise folds
the smooth field at its mid-level, creating sharp, high-contrast cell
BOUNDARIES -- the modular cumulus cells with clean blue gaps. This is the
core of the correction: continuous noise -> billow cells.

CHANGE 2 -- FEATURE SCALES MEASURED FROM noises.png (not guessed)
----------------------------------------------------------------
I inspected the untouched Bliss noises.png (512x512). The cloud channels are
SMOOTH: autocorrelation ~0.5 at lag 16 texels and ~0 at lag 32, i.e. a
correlation length of ~20 texels (the R/G channels are also darker, mean
~0.24, vs B mean ~0.50). Two consequences, both now encoded:
   * The procedural pitch must be  BLISS_NOISE_RES = 512 / 20, so Bliss'
     ORIGINAL coordinate multipliers (worldXZ/20000, /2000, /100000, ...)
     reproduce Bliss' native feature sizes (CloudLarge ~780 blocks,
     CloudSmall ~78 blocks, etc.).
   * The 3D erosion must match the SAME ~20-texel smoothness: densityAtPos
     divides xz by BLISS_NOISE_CORR, making the erosion ~30-block detail.
     Previously it was per-texel (~1.5 block) -- 20x too fine -- which is
     exactly what the vertical ray could not sample and what produced the
     radial "spikes". So the correct scale ALSO removes the spike source.

CHANGE 3 -- CHANNEL LEVELS TUNED OFFLINE FOR THE SCATTERED-CELL LOOK
-------------------------------------------------------------------
A coverage simulation (the exact billow math in numpy) was used to pick the
channel transforms, rather than guessing:
   noiseB (CloudLarge) = value noise, mean ~0.50      (crisp, like B)
   noiseR (CloudSmall) = value noise * 0.85 + dark bias -> mean ~0.30
The dark CloudSmall is essential: the  -(1-CloudSmall)  term drives coverage
NEGATIVE in the gaps, which is what opens the clean blue sky between cells.
At the default coverage slider (0.7) this yields ~58% sky with cloud, ~11%
dense cores and ~21% fully-clear gaps (plus partial-blue between) -- i.e.
scattered cells with gaps, not a sheet.

CHANGE 4 -- CONTRACTS PRESERVED
-------------------------------
   * Depth occlusion (bliss_terrainDist, set in the adapter; tested in the
     cumulus loop AND the altostratus branch of renderLayer) is unchanged.
   * Grazing-angle anti-aliasing is retained: the FINEST cloudVol erosion
     octave (the mix(450,600) Perlin-worley term) is multiplied by
        aa = quintic(clamp((bliss_rayVerticality - BLISS_AA_LO)/(BLISS_AA_HI-BLISS_AA_LO),0,1))
     so it fades out at the horizon (huge steps) and cannot alias into spikes.
   * Noise is still world-anchored (rayProgress includes cameraPosition).

REMOVED (no longer referenced): bliss_fbm2, bliss_fbm3, bliss_coverageField,
and the iteration-4/5 tunables (BLISS_COVER_SCALE/ANISO/COVER_GAIN/EDGE/
ERODE_*/FLUFF_*/BASE_FRAC/TOP_FRAC/ALTO_SCALE/FBM_OCTAVES).

=======================================================================
 DEBUG ITERATION 7 -- VOLUMETRIC TRANSMITTANCE AND SHADING FIX
=======================================================================
SYMPTOM
-------
Iteration 6 shape is correct, but every cloud reads as one flat,
overexposed white mass: tops, bases, near and far layers all bleed
together. No depth, no shadowed undersides, no layer separation.

ROOT CAUSE (it was NOT the lighting math)
-----------------------------------------
bliss_DoCloudLighting + the renderLayer accumulation are verbatim native
Bliss and ARE correct: powder = 1-exp(-10*density); the Beer-Lambert
integration  COLOR += lighting*(1 - exp(-mult*muE)); EXTINCTION *=
exp(-mult*muE)  is intact; self-shadow uses exp(-k*sunShadows). The
problem is the INPUT RADIANCE. Bliss authored those gains for its physical
sky LUT; this port feeds RV's cloudLightColor / cloudAmbientColor through
Bliss' x3.14 (and x2) scatter gains. The forward Mie phase peaks at ~8-10,
so directScattering near the sun is ~25-50x the light colour -- far above
1.0. EVERYTHING clips to white in RV's tonemap, so the real shadow/depth
gradient that the math produces is simply above the white point and
invisible.

FIX 1 -- ANTI-BLOWOUT (bring radiance into a sane HDR range)
------------------------------------------------------------
In bliss_renderClouds the scatter gains are now tunables, much lower than
Bliss' physical-LUT values, and the forward phase peak is capped:
    mieDay = min(mieDay, BLISS_MIE_CLAMP=5.0);     // tames the sun halo
    directScattering      = LightColor*mieDay      * BLISS_SUN_SCATTER  (1.00, was 3.14)
    directMultiScattering = LightColor*mieDayMulti * BLISS_MULTI_SCATTER*2 (0.80, was 3.14)
The ambient (indirect) magnitude is left ALONE, so lit tops keep their
brightness; only the blown-out direct sun term is tamed. Structure that
was clipped to white now survives tonemapping.

FIX 2 -- DENSITY-BASED DEPTH SHADING + BLUE-GREY SHADOWS
-------------------------------------------------------
Added to bliss_DoCloudLighting (after the native direct+indirect terms):
    baseDark   = 1 - BLISS_BASE_DARK*skyScatter;                 // low in layer -> darker
    coreDark   = 1 - BLISS_CORE_DARK*(1 - exp(-3.5*density));    // dense core -> darker (Beer-Lambert)
    depthShade = clamp(baseDark*coreDark, 0, 1);
    shadowCol  = skyLightCol * BLISS_SHADOW_TINT;                // soft blue-grey (0.60,0.68,0.85)
    ambient    = mix(shadowCol, indirectLight, depthShade);
    return directLight*depthShade + ambient;
skyScatter is Bliss' existing bottom->top gradient (clamp((maxHeight-y)/100)),
so this darkens cloud BASES and dense CORES with exponential falloff and
bleeds them toward blue-grey, while lit tops (skyScatter~0, depthShade~1)
stay bright silver-white. That is the white-edge -> grey-blue-core
transition the reference shows.

FIX 3 -- STRONGER SELF-SHADOW FOR LAYER SEPARATION
--------------------------------------------------
The self-shadow absorption is now BLISS_SELFSHADOW=11 (direct) and
*0.27 (~3, multiscatter, unchanged). Bliss' existing cross-layer shadow
(layer1 -> layer0 directLight term) and skylightOcclusion (layer1 dims
layer0's ambient) are untouched, so when an upper deck sits over a lower
one the lower plane now visibly darkens toward blue-grey.

PRESERVED (Iteration 6 advances, NOT touched)
   - the billow abs() coverage, channel transforms and feature scales
     (BLISS_NOISE_CORR / BLISS_NOISE_RES), densityAtPos, cloudCov, cloudVol;
   - depth occlusion (bliss_terrainDist) and grazing-angle AA
     (bliss_rayVerticality fade on the finest erosion octave);
   - world-anchoring of the noise.
Only the lighting/scatter/shading equations changed.

LIGHTING TUNABLES (top of bliss_clouds.glsl)
   BLISS_SUN_SCATTER 1.00  BLISS_MULTI_SCATTER 0.80  BLISS_MIE_CLAMP 5.0
   BLISS_SELFSHADOW 11.0   BLISS_BASE_DARK 0.55      BLISS_CORE_DARK 0.45
   BLISS_SHADOW_TINT (0.60,0.68,0.85)

   - {} () [] balance == 0 on bliss_clouds.glsl                          : PASS
   - #if / #ifdef / #ifndef (32) == #endif (32)                         : PASS

=======================================================================
 DEBUG ITERATION 8 -- EDGE TRANSPARENCY AND TERRAIN SHADOW FIX
=======================================================================
BUG 1 -- DIRTY BLACK OUTLINE ON THIN CLOUDLETS (premultiplied alpha)
--------------------------------------------------------------------
Bliss' renderClouds returns a PREMULTIPLIED (associated) colour: the
front-to-back accumulation
    COLOR += lighting*(1 - exp(-mult*muE)) * TOTAL_EXTINCTION
already folds opacity into COLOR, and the alpha it reports is
1 - TOTAL_EXTINCTION. The correct composite over the sky is therefore
    result = sky*TOTAL_EXTINCTION + COLOR.
But RV composites every cloud with
    color = mix(sky, clouds.rgb, clouds.a) = sky*(1-alpha) + clouds.rgb*alpha
which expects a STRAIGHT (un-premultiplied) colour. The adapter was handing
RV the premultiplied COLOR as if it were straight, so a thin edge
(alpha~0.1) removed ~10% of the sky but added back only COLOR*alpha ~=
straightColour*alpha^2 of light -- almost nothing. Net: the sky darkened
with no cloud light filling in = a dark/black silhouette line around faint
cloudlets.

FIX: un-premultiply in the adapter (the one line that maps Bliss -> RV):
    alpha         = clamp(1 - raw.a, 0, 1);
    straightColor = min(raw.rgb / max(alpha, 1e-3), vec3(8.0));  // /alpha (+ firefly clamp)
    clouds        = vec4(straightColor, alpha);
Now mix() expands to  sky*(1-alpha) + (COLOR/alpha)*alpha = sky*T + COLOR,
the correct premultiplied composite. Thin edges fade to clean translucency
(and thin SUNLIT edges show their silver lining instead of a black rim).
No lighting/shape constants touched.

BUG 2 -- CLOUD SHADOWS NOT PROJECTING ON TERRAIN
------------------------------------------------
Two causes:
  (a) RV's CLOUD_SHADOWS option is commented out by default in common.glsl
      -> no terrain cloud shadows at all. Enabled it (#define CLOUD_SHADOWS).
  (b) With it on, the Bliss style fell through RV's terrain-shadow dispatch
      into the UNBOUND #else branch, which samples noisetex at scale
      0.000002*CLOUD_UNBOUND_SIZE_MULT (~5000-block features) -> one nearly
      uniform shadow patch over the whole visible area = invisible as
      distinct cloud shadows.

FIX: added a dedicated CLOUDS_BLISS branch to the terrain shadow dispatch in
lib/lighting/mainLighting.glsl (alongside the REIMAGINED/gaux4 and UNBOUND
paths). It reproduces the Bliss BILLOW coverage at cloud-cell scale so the
ground darkens under the cumulus cells with cell-sized, moving shadows:
    bcsPos = worldPos.xz + worldPos.y*0.25;  bcsPos.x += syncedTime;
    bLarge = texture(noisetex, bcsPos*0.00006).b;   // ~130-block masses
    bSmall = texture(noisetex, bcsPos*0.0007 +0.37).b; // ~22-block cells
    bCov   = clamp(abs(bLarge*2-1.2)*0.5 - (1-bSmall) + 0.7, 0,1); // billow
    bCov  *= sqrt3(1 - abs(dot(eastVec,lightVec)));  // soften near horizon
    shadowMult *= 1 - 0.80*bCov;
This is texture-based and self-contained (uses only worldPos, syncedTime,
noisetex, eastVec, lightVec -- all already present in every program that
includes mainLighting), so it compiles in the terrain/gbuffers passes
WITHOUT pulling the full cloud renderer (and its cloudColors/sky includes)
into them, which would have caused duplicate-symbol errors. It is a
Bliss-scale APPROXIMATION of the deck (same billow + cell sizes), not a
literal sample of the deferred density field; a literal sample would need
a guarded shadow-only include of the density functions in every terrain
program and is deferred to avoid an untestable cross-program include.

 DEBUG ITERATION 9 -- CLEAR-WEATHER DENSITY, GUI SPEED SYNC, HORIZON FOG
=======================================================================
All three are quantity/scaling tweaks; the Iteration 6 billow shape math and
the Iteration 7/8 radiance + alpha fixes are untouched.

1) HALF THE CLOUDS ON CLEAR DAYS
   rainStrength is 0 on clear days and ramps to 1 in rain/thunder. A shared
   scale is applied to every layer's coverage:
       bliss_clearScale = mix(BLISS_CLEAR_COVERAGE, 1.0, rainStrength);  // clear->0.72, rain->1.0
       LAYER0/1/2_COVERAGE = (original weather mix) * bliss_clearScale;
   This tightens the billow threshold (coverage + LAYER_COVERAGE) only on
   clear days. Offline billow simulation: the default LAYER0_COVERAGE 0.70
   covers ~58% of sky; scaling to 0.70*0.72 ~= 0.50 covers ~28% -> ~0.48x,
   i.e. about half the clouds, with full coverage restored as it rains.
   BLISS_CLEAR_COVERAGE = 1.0 disables the reduction. The scale is shared by
   the self-shadow sampler too, so shadows stay consistent with the clouds.

2) CLOUD SPEED LINKED TO THE GUI SLIDER
   cloud_movement previously used only Bliss' own Cloud_Speed (the RV main
   "Cloud Speed" slider, CLOUD_SPEED_MULT, did nothing for Bliss). Now:
       cloud_movement = (worldTime + ...) / 24 * Cloud_Speed * (CLOUD_SPEED_MULT * 0.01);
   CLOUD_SPEED_MULT is a percent (100 = baseline, range 0..900 in the menu),
   so the in-game slider scales or PAUSES (0) cloud motion. At the defaults
   (Cloud_Speed 1.0, CLOUD_SPEED_MULT 100) the speed equals the old baseline,
   so this is non-regressive; the user's 900 setting now gives 9x drift.
   CLOUD_SPEED_MULT is a global #define from common.glsl, available in every
   program that compiles the cloud code.

3) HORIZON FOG (SOFT DISTANT EXTINCTION)
   The Bliss raymarch has no hard far-cut, so toward the horizon the ray
   reaches the cloud plane at an enormous distance and the deck rendered out
   crisply to infinity (it also is NOT covered by RV's DoFog, which runs
   before clouds are composited in deferred1). Added a distance fade in the
   adapter, on the cloud ALPHA (after the un-premultiply, so colour is
   unaffected):
       cloudDist   = max(CloudLayer0_height - cameraPos.y, 32) / max(|nPlayerPos.y|, 0.05);
       alpha      *= 1.0 - smoothstep(distanceThreshold*BLISS_FOG_START,
                                      distanceThreshold*BLISS_FOG_END, cloudDist);
   distanceThreshold is RV's own per-view-angle cloud render distance (bigger
   looking up, ~far at the horizon), so overhead clouds are untouched and only
   the near-horizon deck dissolves into the haze -- matching how the Unbound /
   Reimagined styles fade. cloudDist is reused for cloudLinearDepth.

   BLISS_FOG_START 0.55  BLISS_FOG_END 1.25   (fractions of cloud render distance)

 DEBUG ITERATION 10 -- HIGH-LAYER ALPHA SMOOTHSTEP + CINEMATIC TIME
                       INTERPOLATION ARCHITECTURE
=======================================================================
NOTE ON THE REFERENCE: the Eclipse package was a Google Drive link that
returned HTTP 403 (auth-gated), so it could not be read. Part 2 below is
built from the standard, well-understood technique for shader time-
smoothing (a per-frame feedback buffer that eases a stored visual time
toward the real worldTime), not from the Eclipse source.

PART 1 -- HIGH-ALTITUDE (ALTOSTRATUS) BLACK-EDGE FIX  [WORKING]
--------------------------------------------------------------
Symptom: grainy, pixelated dark outlines where thin high cirrus meets the
blue sky. Cause: the altostratus is sampled ONCE per pixel (it is a single
plane, not a raymarch loop), so its density 0-crossing -- shape = max(large
- medium*..., 0), returned as shape*shape -- is a hard onset that the
per-pixel ray dither flickers across, alternating cloud/no-cloud at the
edge; with the cloud colour slightly below the bright near-sun sky that
reads as a dark speckled ring.
Fix (bliss_GetAltostratusDensity only -- billow cumulus untouched):
    density  = shape * shape;
    density *= bliss_quintic( clamp(shape * BLISS_ALTO_EDGE, 0.0, 1.0) );
The extra quintic ramp widens the low-density band, so thin edges fade out
over many more blocks: the dither offset now stays inside a gentle gradient
instead of straddling a step, and faint cirrus dissolves into clean
transparency. BLISS_ALTO_EDGE (3.0) tunes the soft-band width.

PART 2 -- CINEMATIC TIME INTERPOLATION  [FRAMEWORK + HOOKS, NON-BREAKING]
------------------------------------------------------------------------
Goal: when worldTime JUMPS (/time set, bed, plugins) the sun/moon/sky/
shadows/clouds should ease to the new time over a configurable duration
with an exponential-out curve, instead of snapping.

Why this is delivered as a framework, not a finished feature: easing a
DISCONTINUITY fundamentally requires per-frame STATE (you must remember last
frame's visual time to step it toward the new target). That state needs a
PERSISTENT feedback buffer (a texel Iris does not clear) plus a prepare-pass
to update it, plus replacing worldTime/sunPosition reads across RV's core
sky + shadow pipeline. Those steps cannot be validated without a live Iris
build, and wiring a persistent buffer + rewiring the shadow matrices blind
would risk the whole (now-working) pack. So this iteration lays down the
complete, correct framework and the activation hooks, kept fully
NON-BREAKING (everything still runs on real time), rather than shipping an
untestable pipeline-wide rewrite.

Delivered and wired:
  * GUI slider  TIME_TRANSITION_SPEED  (seconds, default 1.2, 0 = off) in
    common.glsl, added to the Cloud Settings screen + sliders list + lang.
  * New module  lib/misc/timeInterpolation.glsl  with:
      - eclipseEaseExpOut(dt) = clamp(1 - exp(-dt / TIME_TRANSITION_SPEED), 0,1)
        (the frame-rate-independent exponential-out weight),
      - eclipseAdvanceTime() / eclipseAdvanceVec() (advance a stored value
        toward a target by one frame -- for time, light vectors, sky params),
      - bliss_GetVisualWorldTime() / bliss_GetVisualTimeFract() read interface
        with a SAFE FALLBACK (returns the real game time) so it compiles and
        behaves identically until activated,
      - a fully documented STATE / UPDATE / READ / ACTIVATION architecture
        for the persistent feedback texel.
  * The module is #included in the cloud pipeline (bliss_clouds.glsl) so the
    framework is live there, and the cloud-movement / sun-vector activation
    points are marked with [ECLIPSE TIME HOOK] comments. cloud_movement is a
    file-scope initializer (no user-function calls allowed there on some
    drivers), so it stays on real time until the system is activated inside a
    function body -- documented inline.

To ACTIVATE (the tested follow-up): reserve + persist a feedback texel in
shaders.properties, add the per-frame UPDATE pass, #define ECLIPSE_TIME_ACTIVE
and point bliss_GetVisualWorldTime() at the texel, then swap the worldTime /
sunPosition reads in the sky + shadow code for the bliss_GetVisual* accessors.

PRESERVED (per protocol): Iteration 8 terrain cloud shadows + un-premultiply,
Iteration 9 clear-weather 0.5x coverage + GUI cloud-speed sync + horizon fog,
and the Iteration 6/7 billow shape + radiance. None were touched.

DEBUG ITERATION 11 -- PRODUCTION INTEGRATION OF THE ECLIPSE FEEDBACK
                        EASING SYSTEM (GATED, REAL CODE, SAFE DEFAULT)
=======================================================================
Scope: turn the Iteration 10 time-interpolation FRAMEWORK into a real,
wired feedback system -- behind a master toggle that ships OFF, so the
default pack stays byte-identical to Iteration 10 while the cinematic
path is fully present and one #define away from active.

DESIGN DECISION -- EASE THE SUN VECTOR, NOT A SCALAR CLOCK
----------------------------------------------------------
A smoothed tick-count would force us to re-derive all celestial math from
a fake time and risk wrap/day-boundary errors. Instead we ease the
WORLD-SPACE SUN DIRECTION directly: one persistent texel stores the
already-smoothed sun, and the cloud lighting just reads it. Robust across
midnight, day rollovers and sunPathRotation, and it touches only the
cloud-light domain (the safe place to intervene).

WIRING (all gated by  #define ECLIPSE_TIME_ACTIVE , OFF by default)
-------------------------------------------------------------------
  * common.glsl
      - Master toggle  //#define ECLIPSE_TIME_ACTIVE  (Cloud Settings screen,
        OFF). TIME_TRANSITION_SPEED slider retained (default 1.2 s).
  * lib/pipelineSettings.glsl
      - Gated, REAL code (outside the commented format block):
            const int  colortex15Format = RGBA32F;
            const bool colortex15Clear  = false;   // persistent feedback texel
        Parsed from composite6 + final (the two passes that include
        pipelineSettings), both of which include common.glsl first, so the
        toggle is in scope.
  * lib/uniforms.glsl
      - Gated  uniform sampler2D colortex15;  (the feedback texel).
  * program/composite7.glsl   [THE UPDATE PASS]
      - Gated  /* RENDERTARGETS:3,15 */  (else the untouched DRAWBUFFERS:3).
      - Recomputes the REAL sun with the exact GetSunVector overworld math
        (timeAngle + sunPathRotation -> view -> world), reads last frame's
        stored sun from colortex15, and steps it toward the real sun by
            ew = 1 - exp(-dt / TIME_TRANSITION_SPEED)
        where dt = frameTimeCounter - storedFrameTime. Seeds itself on the
        first frame (no pop), snaps on frameTimeCounter wrap / absurd gaps,
        re-normalises, and stores vec4(smoothedSun, frameTimeCounter).
        Non-overworld dims write vec4(0) (cloud reader falls back to real sun).
  * lib/misc/timeInterpolation.glsl
      - Easing helpers retained (exp-out, exact). READ INTERFACE reworked:
        bliss_GetVisualWorldTime()/...TimeFract() now ALWAYS return real time
        in BOTH modes (link-safe: no forward-decl-without-definition), and the
        active channel is the new
            vec3 bliss_GetVisualSunVec(vec3 realSunVecWorld)
        ON: returns the eased sun from colortex15 (real-sun fallback if the
        texel is unseeded/degenerate). OFF: identity.
  * lib/atmospherics/clouds/bliss_clouds.glsl  [THE READER]
      - Adapter routes the cloud light direction through bliss_GetVisualSunVec
        when ON (and takes sunElevation from the SAME smoothed vector so the
        day/night flip eases in step). The OFF branch (#else) is the verbatim
        Iteration 10 two lines, so OFF is byte-identical.
  * shaders.properties / lang/en_US.lang
      - ECLIPSE_TIME_ACTIVE added to the Cloud Settings screen with a full
        tooltip; TIME_TRANSITION_SPEED tooltip updated (it now gates on the
        master switch). No slider-list change (boolean toggle).

BEHAVIOUR
---------
  OFF (default): clouds light from the real sun exactly as Iteration 10. No
  extra buffer is allocated, no extra render target, no extra sampler. The
  pack is byte-identical.
  ON: a single persistent texel remembers the sun. During normal play the
  real sun barely moves so the lag is invisible; on a time JUMP (/time set,
  sleeping, plugins) the cloud lighting glides across the gap over
  ~TIME_TRANSITION_SPEED seconds with an exponential-out curve instead of
  snapping. 1-frame feedback latency (deferred reads last frame's composite7
  write) -- negligible against a >1 s easing.

DEBUG ITERATION 12 -- SOLVED colortex15 FORMAT TOKEN CRASH AND
                        RELOCATED INTERP CONTROLS TO PERFORMANCE MENU
=======================================================================
Symptom (reported): enabling ECLIPSE_TIME_ACTIVE crashed compilation with
    composite6.fsh: 0(370) : error C1503: undefined variable "RGBA32F"
and the pipeline fell back to vanilla graphics.

ROOT CAUSE
----------
This pack declares NO colortexNFormat constants -- the entire format block in
lib/pipelineSettings.glsl is commented out and RV runs on the host's DEFAULT
buffer formats (the only active format declarations anywhere are the image.*
compute buffers in shaders.properties). The Iteration 11 line
    const int colortex15Format = RGBA32F;
was therefore NOT consumed by the pipeline's format parser; it survived into
the generated GLSL of every pass that includes pipelineSettings (composite6,
final), where the driver hit the bare identifier RGBA32F and aborted -- taking
the whole pipeline down to vanilla. Lower-casing the token would not help: the
token is undefined to the GLSL compiler either way; the directive simply is
not part of this pack's pipeline contract.

FIX 1 -- REMOVE THE FORMAT DECLARATION, ENCODE TO SURVIVE THE DEFAULT FORMAT
---------------------------------------------------------------------------
colortex15 is now left on its default format (and is already persistent --
Iris/OptiFine keep colortex8-15 uncleared by default, exactly what a feedback
texel needs), so NO colortexNFormat / Clear constant is declared at all. The
feedback instead stores its sun direction robustly for any default format:
  * composite7 writes  vec4(sun * 0.5 + 0.5, 1.0)  -- the unit sun ENCODED into
    [0,1] (survives an 8-bit/unsigned default) with a "seeded" flag in .a.
  * the dt for the easing now comes from the  frameTime  uniform (real
    per-frame delta, seconds) instead of a stored frameTimeCounter, so the
    alpha channel is free for the seeded flag and there is no wrap handling.
  * NORMAL PLAY tracks the real sun EXACTLY (the writer snaps when the real sun
    is within ~2 deg of the stored sun; the reader returns the real sun in that
    regime), so there is zero lag and zero low-precision stepping during
    ordinary time-of-day movement. Only a JUMP (>~2 deg gap, i.e. /time set,
    sleeping, plugins) triggers the exponential-out ease; the reader returns
    the eased stored sun until it re-converges, then hands back the real sun.
  * bliss_GetVisualSunVec() decodes [0,1] -> [-1,1], normalises, and falls back
    to the real sun whenever the texel is unseeded or degenerate, so the cloud
    lighting can never go dark.
Net: compiles on both OptiFine and Iris (no exotic tokens, no custom formats),
smooth at the default speed, and graceful if the default buffer is low
precision. When ECLIPSE_TIME_ACTIVE is OFF the pack remains byte-identical --
no buffer, sampler, render target, or directive is added.

FIX 2 -- RELOCATE CONTROLS TO PERFORMANCE, RESTORE THE CLOUDS MENU
------------------------------------------------------------------
  * shaders.properties: removed TIME_TRANSITION_SPEED + ECLIPSE_TIME_ACTIVE
