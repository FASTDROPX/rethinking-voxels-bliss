WHAT THIS BUILD ADDS
--------------------
A new cloud mode "Bliss" (CLOUD_STYLE 2), selectable alongside RV's
existing Reimagined (1) / Unbound (3) / Vanilla (50) / OFF (0). It renders
a 3-layer volumetric cloudscape (two cumulus decks + one high altostratus
sheet) inside RV's pipeline, lit by RV's own cloud light/ambient colours
and exposed through RV's native GetVolumetricClouds() interface so it
appears in the main sky view AND in water/ice/metal reflections with no
reflection-specific wiring.

=======================================================================
 HOW IT PLUGS INTO RV  (interface, unchanged since the first build)
=======================================================================
  shaders/lib/atmospherics/clouds/bliss_clouds.glsl
      - defines GetVolumetricClouds(...) with the EXACT signature RV's
        mainClouds.glsl already calls (so deferred1 + reflections.glsl +
        gbuffers_water + dh_water all drive it).
  shaders/lib/atmospherics/clouds/bliss_clouds_compat.glsl
      - supplies the Bliss #defines / atmosphere constants / light vectors
        that RV does not expose (WsunVec, sunElevation, ray-march counts,
        SKY_GROUND, CloudLayer toggles, etc.).
  shaders/lib/atmospherics/clouds/mainClouds.glsl
      - #ifdef CLOUDS_BLISS -> #include bliss_clouds.glsl
  shaders/lib/common.glsl, shaders.properties, lang/en_US.lang
      - CLOUD_STYLE value 2 = Bliss + the in-game Bliss cloud options
        (per-layer coverage/density/height, Cloud_Speed, Rain_coverage,
         BLISS_CLOUD_BRIGHTNESS).

Light/sky colour is bridged from RV (cloudLightColor / cloudAmbientColor *
BLISS_CLOUD_BRIGHTNESS). Bliss' own physical-sky LUT is NOT ported.

=======================================================================
 NOISE BRIDGE  (foundation; stable since Iteration 2)
=======================================================================
Bliss authored its cloud field against its own 512x512 RGBA noises.png
(3D-from-2D channel trick, hard-coded "/512" texel math). RV ships a
different noisetex (128x128 RGB). Sampling RV's texture with Bliss' texel
math is what caused the original blocky / tiling / moire regressions, so
the noise is reconstructed PROCEDURALLY and self-contained inside
bliss_clouds.glsl (Dave-Hoskins hashes -> quintic value noise -> fbm).
This never tiles and does NOT touch RV's noisetex, so RV's water and other
noise users are unaffected. These helpers are the foundation the density
field below is built on.

 HISTORY (earlier iterations, SUPERSEDED -- kept only as a record)
=======================================================================
 Iter 1: literal port; compiled but rendered broken (wrong noise texture).
 Iter 2: procedural noise bridge -> fixed blocky/tiling/moire, but the
         clouds were a near-uniform full-sky haze.
 Iter 3: bolt-on coverage mask on top of Bliss' original texel-coverage
         math. STILL WRONG: the masses would not stretch, the sky read as
         a chaotic line on the horizon, and the high-frequency erosion
         aliased into radial "spikes" overhead.
 The Iteration 3 approach (keeping Bliss' abs()-billow coverage math and
 patching it) is ABANDONED. Iteration 4 below replaces the whole density
 field. Any claims in earlier logs about iter 2/3 "fixing the shape" are
 withdrawn.

=======================================================================
 DEBUG ITERATION 4 -- DENSITY FIELD REWRITTEN FROM SCRATCH
=======================================================================
WHY THE PREVIOUS NOISE/COVERAGE FAILED TO STRETCH HORIZONTALLY
-------------------------------------------------------------
Bliss' original coverage used a billow transform on a texture read,
    coverage = abs(CloudLarge*2 - 1.2)*0.5 - (1 - CloudSmall)
    layer    = coverage + LAYER_COVERAGE
fed by 2D coords like worldXZ/20000 and worldXZ/2000. Two problems made it
impossible to get sweeping pancakes out of it with procedural noise:

  1. NO LARGE-SCALE TERM. With procedural noise sitting at its 0.5 mean,
     the billow evaluates to a near-constant ~0.3 everywhere -> a uniform
     full-sky sheet. There was nothing controlling WHERE clouds are vs
     open sky, so nothing could "stretch": every direction looked the same
     and at the horizon all of it integrated into a solid line.

  2. EROSION FREQUENCY TOO HIGH. The erosion (densityAtPos) effectively
     ran at ~1.5-block horizontal features. A near-vertical view ray
     crossing a thin layer in ~10 steps cannot sample 1.5-block detail, so
     it aliased into the radial "spikes / thorns" seen overhead. Adding a
     mask (iter 3) did nothing about this.

WHAT WAS CHANGED (exact math)
-----------------------------
The functions bliss_densityAtPos, bliss_cloudCov, bliss_cloudVol and the
old bliss_GetAltostratusDensity were DELETED. The cumulus and altostratus
density are now computed directly:

  * QUINTIC SHAPING CURVE (new bliss_quintic):
        q(x) = x*x*x*(x*(x*6 - 15) + 10)        // 6x^5 - 15x^4 + 10x^3
    Used for EVERY soft ramp / contrast step below. There is no step() and
    no smoothstep() anywhere in the density path -- only this explicit
    quintic. (The value-noise interpolation in bliss_vnoise2/3 also uses
    this exact polynomial on each axis.)

  * HORIZONTAL COVERAGE -- MASSIVE STRETCH (new bliss_coverageField):
        p   = worldXZ * BLISS_COVER_SCALE;   // 0.00045  -> 1 feature ~= 2200 blocks
        p.x *= BLISS_ANISO;                  // 0.55     -> ~4000-block bands along X
        cov = bliss_quintic( bliss_fbm2(p) );// 3 octaves, contrast-shaped
    The key change is BLISS_COVER_SCALE: multiplying world XZ by a tiny
    number stretches one noise feature across thousands of blocks, so the
    masses are huge and sweeping; BLISS_ANISO stretches them further along
    X into elongated pancake bands. The quintic makes them distinct from
    the gaps without a hard threshold.

  * SOFT COVERAGE ONSET (in bliss_GetCumulusDensity), menu-driven:
        layerCov = (layer==1) ? LAYER1_COVERAGE : LAYER0_COVERAGE; // in-game slider
        covAmt   = clamp(layerCov * BLISS_COVER_GAIN, 0, 1);       // 0.7*0.72 ~= 0.5
        d        = (cov - (1 - covAmt)) / BLISS_EDGE;
        cloud    = bliss_quintic( clamp(d, 0, 1) );                // NO hard step
    Below covAmt the field is open sky; above it the quintic ramps cloud in
    smoothly. Lower covAmt (or the in-game coverage slider) = more open sky.

  * VERTICAL VOLUME -- the "don't flatten to a 2D plane" fix:
        h        = (pos.y - minHeight) / (maxHeight - minHeight); // 0..1 in layer
        baseRamp = bliss_quintic( h / BLISS_BASE_FRAC );          // soft flat base
        topTaper = bliss_quintic( (1 - h) / BLISS_TOP_FRAC );     // soft rounded top
        cloud   *= baseRamp * topTaper;
    Density is shaped through the FULL thickness of the layer with smooth
    quintic attenuation strictly from the layer height limits, giving real
    3D volume instead of an infinitely thin sheet on the horizon. The three
    layers stay at their separate heights (250 / 500 / 2000, each 100 thick).

  * EROSION -- LOW FREQUENCY + MULTIPLICATIVE (kills the spikes):
        ep    = pos * BLISS_ERODE_SCALE;  // 0.006 -> ~165-block detail (was ~1.5)
        ep.xz *= 0.5;                      // stretched horizontally too
        er    = bliss_quintic( bliss_fbm3(ep) );
        cloud *= 1 - er * BLISS_ERODE_AMOUNT * (1 - cloud);
    Erosion features are now ~100x larger, so the view ray never
    undersamples them -> no aliased spikes. It is MULTIPLICATIVE and weighted
    by (1-cloud), so it only nibbles cloud EDGES (cores stay solid) and can
    never push density negative (no max()/clamp step needed for it).

  * ALTOSTRATUS (bliss_GetAltostratusDensity) rewritten the same way: one
    very large sweeping bliss_fbm2 field (BLISS_ALTO_SCALE 0.00018) with a
    quintic onset driven by the in-game CloudLayer2 coverage slider.

The result is driven entirely by clear, documented controls instead of
Bliss' texture-tuned constants.

TUNABLES (top of bliss_clouds.glsl)
   BLISS_COVER_SCALE   0.00045  smaller = bigger / more sweeping masses
   BLISS_ANISO         0.55     smaller = longer horizontal pancake bands
   BLISS_COVER_GAIN    0.72     scales the in-game coverage slider (open sky)
   BLISS_EDGE          0.30     soft-onset width (bigger = softer edges)
   BLISS_FBM_OCTAVES   3        coverage detail octaves
   BLISS_ERODE_SCALE   0.006    erosion detail size (do NOT raise -> spikes)
   BLISS_ERODE_AMOUNT  0.55     how deeply erosion bites the edges
   BLISS_BASE_FRAC     0.22     soft base ramp (fraction of layer height)
   BLISS_TOP_FRAC      0.45     soft top taper (fraction of layer height)
   BLISS_ALTO_SCALE    0.00018  altostratus mass size
