WHAT THIS BUILD ADDS
--------------------
A new cloud mode "Bliss" (CLOUD_STYLE 2), selectable alongside RV's
existing Reimagined (1) / Unbound (3) / Vanilla (50) / OFF (0). It renders
a 3-layer volumetric cloudscape (two cumulus decks + one high altostratus
sheet) inside RV's pipeline, lit by RV's own cloud light/ambient colours
and exposed through RV's native GetVolumetricClouds() interface so it
appears in the main sky view AND in water/ice/metal reflections with no
reflection-specific wiring.

=======================================================================
 HOW IT PLUGS INTO RV  (interface, unchanged since the first build)
=======================================================================
  shaders/lib/atmospherics/clouds/bliss_clouds.glsl
      - defines GetVolumetricClouds(...) with the EXACT signature RV's
        mainClouds.glsl already calls (so deferred1 + reflections.glsl +
        gbuffers_water + dh_water all drive it).
  shaders/lib/atmospherics/clouds/bliss_clouds_compat.glsl
      - supplies the Bliss #defines / atmosphere constants / light vectors
        that RV does not expose (WsunVec, sunElevation, ray-march counts,
        SKY_GROUND, CloudLayer toggles, etc.).
  shaders/lib/atmospherics/clouds/mainClouds.glsl
      - #ifdef CLOUDS_BLISS -> #include bliss_clouds.glsl
  shaders/lib/common.glsl, shaders.properties, lang/en_US.lang
      - CLOUD_STYLE value 2 = Bliss + the in-game Bliss cloud options
        (per-layer coverage/density/height, Cloud_Speed, Rain_coverage,
         BLISS_CLOUD_BRIGHTNESS).

Light/sky colour is bridged from RV (cloudLightColor / cloudAmbientColor *
BLISS_CLOUD_BRIGHTNESS). Bliss' own physical-sky LUT is NOT ported.

=======================================================================
 NOISE BRIDGE  (foundation; stable since Iteration 2)
=======================================================================
Bliss authored its cloud field against its own 512x512 RGBA noises.png
(3D-from-2D channel trick, hard-coded "/512" texel math). RV ships a
different noisetex (128x128 RGB). Sampling RV's texture with Bliss' texel
math is what caused the original blocky / tiling / moire regressions, so
the noise is reconstructed PROCEDURALLY and self-contained inside
bliss_clouds.glsl (Dave-Hoskins hashes -> quintic value noise -> fbm).
This never tiles and does NOT touch RV's noisetex, so RV's water and other
noise users are unaffected. These helpers are the foundation the density
field below is built on.

=======================================================================
 DEBUG ITERATION 4 -- DENSITY FIELD REWRITTEN FROM SCRATCH
=======================================================================
WHY THE PREVIOUS NOISE/COVERAGE FAILED TO STRETCH HORIZONTALLY
-------------------------------------------------------------
Bliss' original coverage used a billow transform on a texture read,
    coverage = abs(CloudLarge*2 - 1.2)*0.5 - (1 - CloudSmall)
    layer    = coverage + LAYER_COVERAGE
fed by 2D coords like worldXZ/20000 and worldXZ/2000. Two problems made it
impossible to get sweeping pancakes out of it with procedural noise:

  1. NO LARGE-SCALE TERM. With procedural noise sitting at its 0.5 mean,
     the billow evaluates to a near-constant ~0.3 everywhere -> a uniform
     full-sky sheet. There was nothing controlling WHERE clouds are vs
     open sky, so nothing could "stretch": every direction looked the same
     and at the horizon all of it integrated into a solid line.

  2. EROSION FREQUENCY TOO HIGH. The erosion (densityAtPos) effectively
     ran at ~1.5-block horizontal features. A near-vertical view ray
     crossing a thin layer in ~10 steps cannot sample 1.5-block detail, so
     it aliased into the radial "spikes / thorns" seen overhead. Adding a
     mask (iter 3) did nothing about this.

WHAT WAS CHANGED (exact math)
-----------------------------
The functions bliss_densityAtPos, bliss_cloudCov, bliss_cloudVol and the
old bliss_GetAltostratusDensity were DELETED. The cumulus and altostratus
density are now computed directly:

  * QUINTIC SHAPING CURVE (new bliss_quintic):
        q(x) = x*x*x*(x*(x*6 - 15) + 10)        // 6x^5 - 15x^4 + 10x^3
    Used for EVERY soft ramp / contrast step below. There is no step() and
    no smoothstep() anywhere in the density path -- only this explicit
    quintic. (The value-noise interpolation in bliss_vnoise2/3 also uses
    this exact polynomial on each axis.)

  * HORIZONTAL COVERAGE -- MASSIVE STRETCH (new bliss_coverageField):
        p   = worldXZ * BLISS_COVER_SCALE;   // 0.00045  -> 1 feature ~= 2200 blocks
        p.x *= BLISS_ANISO;                  // 0.55     -> ~4000-block bands along X
        cov = bliss_quintic( bliss_fbm2(p) );// 3 octaves, contrast-shaped
    The key change is BLISS_COVER_SCALE: multiplying world XZ by a tiny
    number stretches one noise feature across thousands of blocks, so the
    masses are huge and sweeping; BLISS_ANISO stretches them further along
    X into elongated pancake bands. The quintic makes them distinct from
    the gaps without a hard threshold.

  * SOFT COVERAGE ONSET (in bliss_GetCumulusDensity), menu-driven:
        layerCov = (layer==1) ? LAYER1_COVERAGE : LAYER0_COVERAGE; // in-game slider
        covAmt   = clamp(layerCov * BLISS_COVER_GAIN, 0, 1);       // 0.7*0.72 ~= 0.5
        d        = (cov - (1 - covAmt)) / BLISS_EDGE;
        cloud    = bliss_quintic( clamp(d, 0, 1) );                // NO hard step
    Below covAmt the field is open sky; above it the quintic ramps cloud in
    smoothly. Lower covAmt (or the in-game coverage slider) = more open sky.

  * VERTICAL VOLUME -- the "don't flatten to a 2D plane" fix:
        h        = (pos.y - minHeight) / (maxHeight - minHeight); // 0..1 in layer
        baseRamp = bliss_quintic( h / BLISS_BASE_FRAC );          // soft flat base
        topTaper = bliss_quintic( (1 - h) / BLISS_TOP_FRAC );     // soft rounded top
        cloud   *= baseRamp * topTaper;
    Density is shaped through the FULL thickness of the layer with smooth
    quintic attenuation strictly from the layer height limits, giving real
    3D volume instead of an infinitely thin sheet on the horizon. The three
    layers stay at their separate heights (250 / 500 / 2000, each 100 thick).

  * EROSION -- LOW FREQUENCY + MULTIPLICATIVE (kills the spikes):
        ep    = pos * BLISS_ERODE_SCALE;  // 0.006 -> ~165-block detail (was ~1.5)
        ep.xz *= 0.5;                      // stretched horizontally too
        er    = bliss_quintic( bliss_fbm3(ep) );
        cloud *= 1 - er * BLISS_ERODE_AMOUNT * (1 - cloud);
    Erosion features are now ~100x larger, so the view ray never
    undersamples them -> no aliased spikes. It is MULTIPLICATIVE and weighted
    by (1-cloud), so it only nibbles cloud EDGES (cores stay solid) and can
    never push density negative (no max()/clamp step needed for it).

  * ALTOSTRATUS (bliss_GetAltostratusDensity) rewritten the same way: one
    very large sweeping bliss_fbm2 field (BLISS_ALTO_SCALE 0.00018) with a
    quintic onset driven by the in-game CloudLayer2 coverage slider.

The result is driven entirely by clear, documented controls instead of
Bliss' texture-tuned constants.

TUNABLES (top of bliss_clouds.glsl)
   BLISS_COVER_SCALE   0.00045  smaller = bigger / more sweeping masses
   BLISS_ANISO         0.55     smaller = longer horizontal pancake bands
   BLISS_COVER_GAIN    0.72     scales the in-game coverage slider (open sky)
   BLISS_EDGE          0.30     soft-onset width (bigger = softer edges)
   BLISS_FBM_OCTAVES   3        coverage detail octaves
   BLISS_ERODE_SCALE   0.006    erosion detail size (do NOT raise -> spikes)
   BLISS_ERODE_AMOUNT  0.55     how deeply erosion bites the edges
   BLISS_BASE_FRAC     0.22     soft base ramp (fraction of layer height)
   BLISS_TOP_FRAC      0.45     soft top taper (fraction of layer height)
   BLISS_ALTO_SCALE    0.00018  altostratus mass size

=======================================================================
 DEBUG ITERATION 5 -- depth occlusion, fluff, world-anchor verification
=======================================================================
The Iteration 4 rewrite removed the spikes but exposed/!left three issues.
Each is addressed below; all changes are inside bliss_clouds.glsl.

(1) "CLOUDS FOLLOW THE CAMERA" -- diagnosed, root cause was NOT view-space
-------------------------------------------------------------------------
Checked the coordinate chain end to end:
    bliss_layerStartingPosition():
        position = dV_view*dither + cameraPos + (dV_view/abs(dV_view.y))*flip
    renderLayer(): rayProgress starts there and does rayProgress += dV_view
    GetCumulusDensity()/coverageField() sample pos.xz == rayProgress.xz
cameraPos IS cameraPosition (absolute world), so the sampled position is
absolute WORLD space already -- the noise is world-anchored, not camera
space. Injecting cameraPosition again would DOUBLE it and actually break
anchoring, so that was not done.

The real reason it LOOKED locked: BLISS_COVER_SCALE was 0.00045, i.e. cloud
features ~2200 blocks wide. Moving/looking shifts the sampled noise by a
tiny fraction of a feature, so the near-uniform field appears static. Fix:
   - BLISS_COVER_SCALE 0.00045 -> 0.0010  (~1000-block masses: real shape,
     visible motion parallax as the player moves)
   - BLISS_ANISO       0.55    -> 0.70    (rounder masses, less "tube")
   - plus the fluff pass in (3), whose fine detail makes the world-anchored
     parallax obvious.

(2) CLOUDS DRAWING THROUGH TERRAIN / MOUNTAINS -- real bug, now fixed
--------------------------------------------------------------------
Bliss' terrain-intersection test was compiled out (CLOUDS_INTERSECT_TERRAIN
undefined) AND it derived its cutoff from length(FragPosition), which in
this port is a normalized direction * 1024 (a constant ~1024) -- useless.
So nothing stopped the raymarch at solid geometry and clouds painted over
distant mountains.

RV already provides the right value: lViewPosM = world distance to the
solid fragment on this pixel (it is a parameter of GetVolumetricClouds and
is how RV's own Reimagined/Unbound clouds occlude -- see reimaginedClouds
lines "if (lTracePos > lViewPosM)"). Wiring:
   - new runtime global  float bliss_terrainDist;
   - adapter sets it:  bliss_terrainDist = (skyFade >= 0.7) ? 1e9 : lViewPosM;
       * skyFade >= 0.7 is RV's own "this pixel is sky" threshold -> 1e9 so
         distant horizon clouds are NOT culled.
       * otherwise it is the real terrain distance.
   - renderLayer (both the cumulus loop and the altostratus branch) now does
         IntersecTerrain = length(rayProgress - cameraPosition) > bliss_terrainDist;
     and breaks/returns when true. The old #ifdef CLOUDS_INTERSECT_TERRAIN
     blocks and the bogus length(FragPosition) cutoff were deleted.
Result: on terrain pixels the ray stops at the terrain depth, so clouds no
longer render over or through blocks/mountains; on sky pixels they render
to the horizon as before.

(3) "SMOOTH PLASTIC" -- added a high-frequency fluff pass (anti-aliased)
-----------------------------------------------------------------------
The 165-block erosion alone left smooth, synthetic surfaces. Added a second
detail pass in bliss_GetCumulusDensity (LoD>=1 only):
    fp     = pos * BLISS_FLUFF_SCALE; fp.xz *= 0.6;          // ~22-76 block detail
    fluff  = bliss_vnoise3(fp)*0.65 + bliss_vnoise3(fp*2 + 7.3)*0.35; // 2 octaves
    cloud *= 1 - bliss_quintic(fluff) * BLISS_FLUFF_AMOUNT * fluffFade * (1 - cloud);
Two things keep this from bringing the spikes back:
   - it is MULTIPLICATIVE and weighted by (1-cloud), so it only nibbles
     EDGES (cores stay solid) and can never go negative; and
   - fluffFade = quintic(clamp((bliss_rayVerticality - 0.30)/0.40, 0, 1)).
     bliss_rayVerticality = abs(view ray .y) is set in renderClouds. The
     fluff is therefore full strength only when looking up (small, densely
     sampled ray steps) and fades to ZERO toward the horizon (huge steps),
     which is precisely where fine detail used to alias into spikes. Fine
     detail is added exactly where the ray can resolve it.

NEW / CHANGED TUNABLES
   BLISS_COVER_SCALE  0.00045 -> 0.0010   bigger-but-shaped masses + parallax
   BLISS_ANISO        0.55    -> 0.70      rounder, less tube-like
   BLISS_FLUFF_SCALE  (new) 0.022          micro-detail size
   BLISS_FLUFF_AMOUNT (new) 0.55           fluff bite depth
   runtime globals: bliss_terrainDist, bliss_rayVerticality

=======================================================================
 DEBUG ITERATION 6 -- BILLOW CONTRAST MATH RESTORED (cells, not sheets)
=======================================================================
WHAT WAS WRONG
--------------
Iteration 4/5 replaced Bliss' coverage with a smooth, CONTINUOUS quintic
onset of a low-frequency fbm. Continuous noise + a soft onset can only ever
produce smooth, sinuous SHEETS -- there is no mechanism to fold the field
into discrete cells. On top of that the feature scale was guessed far too
large (~1000-block masses), so within any view the field barely varied =
"plastic". Both are fixed here.

CHANGE 1 -- RESTORED THE BLISS BILLOW COVERAGE (the abs() contrast math)
-----------------------------------------------------------------------
The smooth onset / vertical-profile / coverageField functions were DELETED
and Bliss' ORIGINAL density field was put back VERBATIM:
    bliss_densityAtPos, bliss_cloudCov, bliss_cloudVol, bliss_GetCumulusDensity
    bliss_GetAltostratusDensity
Coverage is once again the BILLOW form
    layer0:  abs(CloudLarge*2.0 - 1.2)*0.5 - (1.0 - CloudSmall) + LAYER0_COVERAGE
    layer1:  abs(CloudLarge - 0.8)         -        CloudSmall  + LAYER1_COVERAGE - 0.5
plus Bliss' Topshape/Baseshape vertical erosion and the Perlin-worley
cloudVol erosion. Taking the ABSOLUTE value of the (inverted) noise folds
the smooth field at its mid-level, creating sharp, high-contrast cell
BOUNDARIES -- the modular cumulus cells with clean blue gaps. This is the
core of the correction: continuous noise -> billow cells.

CHANGE 2 -- FEATURE SCALES MEASURED FROM noises.png (not guessed)
----------------------------------------------------------------
I inspected the untouched Bliss noises.png (512x512). The cloud channels are
SMOOTH: autocorrelation ~0.5 at lag 16 texels and ~0 at lag 32, i.e. a
correlation length of ~20 texels (the R/G channels are also darker, mean
~0.24, vs B mean ~0.50). Two consequences, both now encoded:
   * The procedural pitch must be  BLISS_NOISE_RES = 512 / 20, so Bliss'
     ORIGINAL coordinate multipliers (worldXZ/20000, /2000, /100000, ...)
     reproduce Bliss' native feature sizes (CloudLarge ~780 blocks,
     CloudSmall ~78 blocks, etc.).
   * The 3D erosion must match the SAME ~20-texel smoothness: densityAtPos
     divides xz by BLISS_NOISE_CORR, making the erosion ~30-block detail.
     Previously it was per-texel (~1.5 block) -- 20x too fine -- which is
     exactly what the vertical ray could not sample and what produced the
     radial "spikes". So the correct scale ALSO removes the spike source.

CHANGE 3 -- CHANNEL LEVELS TUNED OFFLINE FOR THE SCATTERED-CELL LOOK
-------------------------------------------------------------------
A coverage simulation (the exact billow math in numpy) was used to pick the
channel transforms, rather than guessing:
   noiseB (CloudLarge) = value noise, mean ~0.50      (crisp, like B)
   noiseR (CloudSmall) = value noise * 0.85 + dark bias -> mean ~0.30
The dark CloudSmall is essential: the  -(1-CloudSmall)  term drives coverage
NEGATIVE in the gaps, which is what opens the clean blue sky between cells.
At the default coverage slider (0.7) this yields ~58% sky with cloud, ~11%
dense cores and ~21% fully-clear gaps (plus partial-blue between) -- i.e.
scattered cells with gaps, not a sheet.

CHANGE 4 -- CONTRACTS PRESERVED
-------------------------------
   * Depth occlusion (bliss_terrainDist, set in the adapter; tested in the
     cumulus loop AND the altostratus branch of renderLayer) is unchanged.
   * Grazing-angle anti-aliasing is retained: the FINEST cloudVol erosion
     octave (the mix(450,600) Perlin-worley term) is multiplied by
        aa = quintic(clamp((bliss_rayVerticality - BLISS_AA_LO)/(BLISS_AA_HI-BLISS_AA_LO),0,1))
     so it fades out at the horizon (huge steps) and cannot alias into spikes.
   * Noise is still world-anchored (rayProgress includes cameraPosition).

REMOVED (no longer referenced): bliss_fbm2, bliss_fbm3, bliss_coverageField,
and the iteration-4/5 tunables (BLISS_COVER_SCALE/ANISO/COVER_GAIN/EDGE/
ERODE_*/FLUFF_*/BASE_FRAC/TOP_FRAC/ALTO_SCALE/FBM_OCTAVES).
